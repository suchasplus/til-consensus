# Adjudication 裁决模式

## 模式详述

`adjudication`（裁决模式）是 `til-consensus` 的**默认模式**，也是系统内对逻辑严谨度、证据追溯度要求最高的工作流。

**核心逻辑：**
它模拟了一个基于证据驱动的受控法庭体系。它的设计理念认为：让大模型直接生成一篇长篇大论作为最终答案往往是不可靠且难以验证的。因此，裁决模式强制**将观点拆解为“原子化”的主张（Claim）**，并围绕这些独立的原子发起对抗、寻找证据进行验证，最终由中立的法官逐条宣判。

**适用场景：**
- 验证代码 Patch 是否真正修复了竞态问题或 Bug。
- 确认 Benchmark 报告中的性能主张是否成立。
- 架构设计决策的逻辑推演与证明。
- 任何需要高置信度结论，且结论必须有铁证支撑的场景。

**固定宏观阶段（Phase）：**
1. `frame`: 把模糊的自然语言请求转化为结构化的案件清单（Case Manifest）。
2. `ingest`: 从外部或本地摄入上下文和静态材料作为证据入账。
3. `propose`: 主张方（Proposer）提取并提出原子化的 Claim。
4. `challenge`: 反驳方（Challenger）针对具体 Claim 寻找漏洞发起攻击。
5. `verify`: 验证方（Verifier）尝试用客观确定性方法或语义模型去寻找证据证明/伪证 Claim。
6. `revise`: **强制动作**。主张方必须针对被挑战和验证的结果，对其 Claim 进行修改或撤回。
7. `adjudicate`: 法官（Arbiter）根据完整的证据链，逐条宣判每条 Claim 的去留。
8. `report`: 产出最终结果报告。
9. `action`: 基于系统的风险门禁，执行受控的后续动作。
10. `observe`: 执行后验观测（如运行健康检查脚本），验证真实结果。如果有矛盾证据，可能触发 Reopen。

## 核心名词词汇表

| 英文单词 | 中文翻译 | 场景和用途 |
| :--- | :--- | :--- |
| **Claim** | 主张 / 论点 | 裁决的“一等公民”。是一句原子化的观点，不是大段文本。所有的攻击、验证和裁决都必须精准挂载（Target）到某条 Claim 上。 |
| **Proposer** | 提出者 / 主张方 | 角色。负责在 `propose` 阶段，根据题意提出初始的 Claim 集合。 |
| **Challenger / Attack** | 挑战者 / 攻击记录 | 角色/动作。负责在 `challenge` 阶段找出 Claim 的漏洞。攻击必须指明严重程度（Severity）和建议的证伪方法。 |
| **Verifier** | 验证者 | 角色。负责在 `verify` 阶段找证据。分为 Deterministic（优先，如跑脚本/查日志）和 Semantic（大模型语义兜底）。 |
| **Revise** | 修订 | 动作。在面临挑战或验证结果后，Proposer 必须履行的强制动作：对 Claim 选择修正、降级置信度、撤回或标记未决。 |
| **Arbiter** | 裁决者 / 法官 | 角色。整个模式的核心决策节点。中立角色，不提新观点，而是基于证据链对每条 Claim 给出最终判决。 |
| **Disposition** | 处置结果 | Arbiter 给出的单条 Claim 的最终判决状态，例如：`keep`（保留）、`reject`（驳回）、`unresolved`（未决）。 |
| **Caveat** | 附加条件 / 保留意见 | 当一条 Claim 大体正确但有瑕疵时，Disposition 会判定为 `keep_with_caveat`，并附带此说明（例如：“仅在 Linux 环境下成立”）。 |
| **Case Manifest** | 案件清单 / 任务清单 | 在 `frame` 阶段生成，将模糊任务转化为明确的成功标准、风险等级和准入条件，框定后续所有辩论的边界。 |
